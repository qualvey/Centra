# EventGuard - Architecture & Development Plan

## Project Vision

EventGuard is a modular event analysis and automatic response framework written in Go.

It is **not** a Sing-box-specific tool.

The long-term goal is to build a generic event processing engine capable of receiving logs from different services, converting them into unified events, evaluating them through a rule engine, and executing configurable actions.

Core pipeline:

```
Log
    ↓
Reader
    ↓
Parser
    ↓
Event
    ↓
Rule Engine
    ↓
Action
```

The project should remain modular, extensible and maintainable as it grows.

---

# Design Principles

## 1. Everything starts with an Event

Logs are only input.

Every parser must convert logs into a unified Event object.

No module except Parser should process raw log strings.

Example:

```
Sing-box Log
        ↓
Sing-box Parser
        ↓
Event
```

```
SSH Log
      ↓
SSH Parser
      ↓
Event
```

```
Nginx Log
        ↓
Nginx Parser
        ↓
Event
```

All downstream components work only with Event.

---

## 2. Single Responsibility

Every component has exactly one responsibility.

Reader

* Read logs
* Follow journalctl or files
* No parsing
* No analysis

Parser

* Convert logs into Event
* No statistics
* No blocking
* No firewall operations

Rule Engine

* Analyze events
* Evaluate rules
* Calculate reputation
* Produce decisions

Action

* Execute decisions
* Firewall
* Notifications
* Webhook
* Reports

Storage

* Persist events
* Persist IP state
* Provide query interface

---

## 3. Interface Driven Design

Core modules must communicate through interfaces.

New implementations should be added without modifying existing modules.

Example:

```
Reader
Parser
Rule
Action
Storage
```

Adding a new parser should never require changing the Rule Engine.

---

# Data Model

## Event

Event is immutable.

It represents something that happened.

Typical fields:

* Timestamp
* Source
* Service
* EventType
* Severity
* Remote IP
* Message
* Metadata

Examples:

```
Source:
sing-box

EventType:
reality_invalid

IP:
45.227.254.152
```

```
Source:
ssh

EventType:
password_failed
```

---

# Storage Design

Storage is divided into two logical parts.

## Event Database

Stores immutable historical events.

Each event is permanently recorded.

Purpose:

* Audit
* Statistics
* Trend analysis
* Rule debugging
* Historical analysis

Example:

| Time  | IP             | Event           |
| ----- | -------------- | --------------- |
| 21:13 | 45.227.254.152 | REALITY Invalid |
| 21:30 | 45.227.254.152 | REALITY Invalid |

Historical events must never be modified.

---

## IP State Database

Stores current state for each IP.

Example fields:

* IP
* Score
* Count
* LastSeen
* Status

Status example:

```
Observed

↓

Suspicious

↓

Blocked

↓

Expired
```

Unlike Event Database, this table is mutable.

It represents current reputation rather than historical facts.

---

# Facts vs Decisions

This project separates facts from decisions.

Facts

* Event happened
* Timestamp
* Source
* IP

Facts never change.

Decision

* Block
* Ignore
* Alert
* Expire

Decisions are generated by Rule Engine based on facts.

Changing rules should not require changing historical data.

---

# Rule Engine

The Rule Engine consumes Events.

It evaluates configurable rules.

Example:

```
REALITY Invalid

↓

Within 5 minutes

↓

More than 3 times

↓

Trigger
```

Rules never execute firewall operations directly.

Rules only produce decisions.

---

# Reputation System

Every IP has a reputation score.

Example:

```
REALITY Invalid
+20

SSH Password Failed
+10

HTTP Scan
+15

Port Scan
+30
```

Scores may decay over time.

Blocking decisions are based on score thresholds.

---

# Action System

Actions execute Rule Engine decisions.

Examples:

* nftables
* iptables
* ipset
* Notification
* Webhook
* Report

Rule Engine should never know how Actions are implemented.

Example:

```
Rule

↓

Decision

↓

Firewall Action
```

Firewall implementation must be replaceable.

---

# First Release (MVP)

Supported source:

* Sing-box

Features:

* Read journalctl
* Parse Sing-box logs
* Generate Events
* Count events by IP
* Output block recommendations

Example:

```
Need Block

IP:
45.227.254.152

Reason:
REALITY Invalid

Count:
5
```

No automatic firewall operations.

No reputation lookup.

No web UI.

Focus on architecture first.

---

# Second Release

Add:

* nftables integration
* Whitelist
* Blocklist
* Automatic blocking
* Automatic unblock
* Configurable thresholds

---

# Third Release

Support more log sources.

Examples:

* SSH
* Nginx
* Docker
* Systemd services

Each source implements its own Parser.

Existing modules should remain unchanged.

---

# Future Extensions

Possible future modules:

* Reputation Provider
* ASN Detection
* CIDR Aggregation
* GeoIP
* Threat Intelligence
* Metrics Export
* Prometheus
* REST API
* Web Dashboard

These features should be implemented as independent modules whenever possible.

---

# Development Philosophy

The project should prioritize architecture over features.

When adding functionality:

Prefer:

* adding new modules
* implementing interfaces
* extending through composition

Avoid:

* modifying existing modules
* adding special cases
* creating large central objects
* coupling Parser with Rule Engine
* coupling Rule Engine with firewall implementation

As the project grows, complexity should remain localized inside individual modules rather than spreading throughout the entire codebase.

The long-term objective is to build a reusable event processing framework, not merely a Sing-box log parser.
